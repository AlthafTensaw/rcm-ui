# Accessibility

Every component in `@tensaw/design-system` ships with WCAG 2.1 AA-level
accessibility built in. This doc is the contract: what we promise, what we
test, and the patterns consumers must follow to keep that promise when they
compose components.

The design system can't make an inaccessible app accessible ã composition
matters. But it can make accessible apps the path of least resistance, and
that's the bar.

---

## What every component guarantees

1. **Keyboard reachable.** Every interactive control has a visible focus
   ring (`focus-visible:ring-2 focus-visible:ring-ring`) and responds to
   the keyboard idioms its role implies (Enter on buttons, arrow keys on
   menus, Tab order in forms).
2. **Screen-reader name.** Either via visible text content, an `aria-label`
   prop, or a `<Label>` association. Icon-only components require
   `aria-label` at the type level.
3. **Role + state semantics.** We lean on Radix UI primitives where roles
   are non-trivial (Dialog, Menu, Tabs, Tooltip). Native HTML where roles
   are trivial (Button, Input, Checkbox).
4. **Color contrast ãË 4.5:1** for body text against its background, ãË 3:1
   for large text and graphical elements. Token values in
   `DESIGN_TOKENS.md` are picked with this constraint.
5. **No keyboard traps.** Modals trap focus; trapping ends on close.
   Dropdowns return focus to their trigger on close.
6. **Reduced motion.** Animations respect
   `@media (prefers-reduced-motion: reduce)` where Radix supplies it.

## The component-by-component checklist

| Concern | Component handles | Consumer handles |
| --- | --- | --- |
| Accessible name | `<Label>` + `htmlFor`, `aria-label` enforcement on icon-only controls | Pass meaningful labels; don't reuse the same label for multiple controls |
| Focus ring | `focus-visible:ring-*` on every interactive surface | Don't override with custom outlines that fail contrast |
| Keyboard activation | Enter/Space on Button; arrow keys on RadioGroup, Tabs, Menu | Don't intercept keydown to add custom behavior that breaks defaults |
| Form errors | `aria-invalid` + `aria-describedby` to error message id | Surface real validation messages, not just "Invalid input" |
| Modal focus trap | Radix Dialog handles trap + initial focus + restoration | Don't manually focus an element inside a Dialog on mount |
| Dropdown semantics | Radix Menu sets `role="menu"` + `aria-orientation` | Use `<DropdownMenuItem>` for items; don't inline arbitrary `<button>`s |
| Tooltip | Radix Tooltip handles delay, dismiss, and `aria-describedby` | Don't use Tooltip for critical-path content; keep it supplementary |
| Loading announcements | `aria-busy="true"` + `<Spinner>`'s `<span class="sr-only">Loading</span>` | Provide a meaningful operation-in-progress message via `loadingLabel` when default doesn't fit |
| Toast queue | ToastHost wraps in `role="region" aria-label="Notifications"` | Push toasts with concrete `title`s, not "Done" |

## Testing accessibility

### In unit tests

Every interactive component's test file uses `@testing-library/react`'s
semantic queries ã `getByRole`, `getByLabelText`, `getByText`. If a query
needs `getByTestId`, the component's accessibility is suspect.

```tsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

it('opens the menu via keyboard', async () => {
  const user = userEvent.setup();
  render(<DropdownMenu trigger={<button>Open</button>}>ãÎ</DropdownMenu>);
  await user.tab();
  await user.keyboard('{Enter}');
  expect(await screen.findByRole('menu')).toBeDefined();
});
```

### In Storybook

Every story has the `@storybook/addon-a11y` panel enabled
(`storybook/.storybook/preview.tsx` configures it globally with
`color-contrast` rule on). Open any story ã A11y panel ã verify there are
no violations. New stories are expected to be clean before merge.

### In integration tests

The patient app's integration tests use `axe-core` via
`@axe-core/playwright` (Phase 13 work) to run a full accessibility audit
on each page in the test suite.

## Common pitfalls

### Disabling buttons during loading

```tsx
// ã DON'T
<Button disabled={isSaving}>Save</Button>

// ã DO
<Button loading={isSaving}>Save</Button>
```

`loading` adds `aria-busy="true"` and announces the in-flight state. A
plain `disabled` doesn't tell the user anything is happening ã it just
greys out.

### Stripping focus styles to "make it look cleaner"

```tsx
// ã DON'T
<Button className="focus:outline-none">ãÎ</Button>
```

The visible focus ring is non-negotiable. If it conflicts with the design,
the design is wrong. Reach for `focus-visible:ring-offset-2` to add space
around it; don't remove it.

### Using divs as buttons

```tsx
// ã DON'T
<div onClick={handleClick} className="cursor-pointer rounded bg-primary p-2">
  Save
</div>

// ã DO
<Button onClick={handleClick}>Save</Button>
```

Divs aren't keyboard-reachable, don't announce their role to assistive
tech, and don't pick up form submit semantics. Use `<Button>` (or
`<Link>` for navigation).

### Adding tooltips to make controls discoverable

```tsx
// ã DON'T (tooltip-only labelling)
<Tooltip content="Delete">
  <IconButton icon={<Trash2 />} />
</Tooltip>

// ã DO
<Tooltip content="Delete claim and dependent records">
  <IconButton aria-label="Delete claim" icon={<Trash2 />} />
</Tooltip>
```

Tooltips are supplementary. The control still needs an accessible name
that doesn't depend on hover. The IconButton's enforced `aria-label` makes
this a TypeScript error if you forget.

### Custom modals without a focus trap

```tsx
// ã DON'T build your own Dialog
{open && <div className="fixed inset-0 ãÎ">ãÎ</div>}

// ã DO use the Dialog primitive
<Dialog open={open} onOpenChange={setOpen} title="ãÎ">ãÎ</Dialog>
```

Radix Dialog handles focus trap, initial focus on the first interactive
element, restoration to the trigger on close, and Escape-to-dismiss.
Recreating these is risky and rarely worth it.

### Misusing role attributes

```tsx
// ã DON'T
<div role="button" tabIndex={0} onClick={ãÎ}>

// ã DO
<Button>ãÎ</Button>
```

`role="button"` on a div needs you to also handle Enter and Space
manually, plus `aria-pressed` if it's a toggle, plus a focus ring.
`<Button>` does all of this.

### Putting interactive content inside a Tooltip

Tooltips are read-only by design. Don't render a button or link inside
Tooltip content ã keyboard users can't focus through the tooltip. If the
content needs interaction, use Popover instead.

### Insufficient contrast on muted text

`text-muted-foreground` against `bg-background` is the lightest combination
in the system at the AA threshold. Don't compose it with any other muted
background ã `text-muted-foreground` on `bg-muted` is below 4.5:1. Use
`text-foreground` instead.

## When you find an accessibility bug

1. Check whether the component already supports the affordance you're
   missing ã many `aria-*` props are forwarded through `...rest`.
2. If not, file an issue *before* working around it. Drift between
   "what the component should do" and "what consumers do to compensate" is
   a maintainability tax.
3. If it's a Radix-level concern (focus trap, role, keyboard pattern), it's
   probably worth filing upstream.
